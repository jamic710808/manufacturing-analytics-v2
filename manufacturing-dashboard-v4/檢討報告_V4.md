## 二、出錯與修復歷程

### Phase 1：React 18 / React 17 API 混淆（最關鍵錯誤）

**問題**：`ReactDOM.render is not a function` 頁面空白

**根本原因**：`single-spa-react` v4.6.1 內部呼叫 `ReactDOM.render()`（React 17 API），但所有 entry point 錯誤使用 `react-dom/client`（React 18 `createRoot` API）

| 錯誤匯入 | 正確匯入 |
|---------|---------|
| `import ReactDOM from 'react-dom/client'` | `import ReactDOM from 'react-dom'` |

**受影響檔案**（全部 4 個 entry points）：
- `packages/page-overview/src/index.tsx`
- `packages/page-inventory/src/index.tsx`
- `packages/page-procurement/src/index.tsx`
- `packages/page-supplier/src/index.tsx`

**修復方式**：將 `import ReactDOM from 'react-dom/client'` 改為 `import ReactDOM from 'react-dom'`

**Why this happened**：未注意到 Single-SPA 框架與 React 18 的 API 相容性問題。`single-spa-react` v4.6.1 鎖定的是 React 17 的 API，無法直接使用 React 18 的 `createRoot`。

---

### Phase 2：Vite 記憶體 bundling 誤解

**問題**：驗證修復時懷疑 Vite dev server 有問題

`curl localhost:9001/dist/spa.js` 回傳 9.6 MB（記憶體中的完整 bundle），磁碟上的 `dist/spa.js` 只有 1.2 MB。這是 **Vite dev 模式的正常行為**，並非錯誤。

**驗證方式**：使用 Chrome DevTools MCP 直接截取瀏覽器 snapshot，確認 JS 是否正確執行，而非依賴磁碟檔案大小。

---

### Phase 3：開發伺服器 Port 衝突

**問題**：Ports 9000–9028 全部被佔用

**現象**：Shell 啟動在 9029，4 個 page 微前端分別在 9001–9004

**修復**：手動指定不同 port，確認所有服務正常運行

## 三、已發現但未修復的 Bug

### Bug 1：`Duplicate key "name"` 警告（InventoryPage.tsx:66）

```tsx
// 第 66 行 — 錯誤
{ name: '原料', value: 4200000 },
{ name: '半成品', value: 3100000 },
{ name: '成品', value: 2800000 },
{ name: '包裝材', value: 1200000 },
{ name: '間接物料', name: '間接物料', value: 1268000 }, // ← 重複 key
```

**說明**：陣列中倒數第二個物件有 `name` 重複賦值（物件內同時有 `name: '間接物料'` 作為第一個屬性，又有一個 `name: '間接物料'`）。這導致 React 渲染時出現 `Warning: Encountered two children with the same key`。

**影響**：不影響功能運行，但 Console 會有警告，長時間執行可能影響效能。

## 七、核心教訓

1. **規格先於實作**：在定義 36 個導航項目之前，應先確認實作能量是否能跟上。否則會造成「承諾過度」的錯誤印象。

2. **Framework API 相容性**：使用 Single-SPA 框架時，必須確認 `single-spa-react` 與 React 版本是否相容。正確的做法是查閱 `single-spa-react` 的 peer dependencies，而非假設新版 React 向後相容。

3. **Dev vs Production 行為差異**：Vite dev 模式將所有依賴 bundling 到記憶體（~9.6 MB），磁碟上的檔案（~1.2 MB）僅用於 production build。驗證時應使用瀏覽器 snapshot，而非磁碟檔案大小。

4. **分階段交付**：建議未來採用「核心功能優先」，先交付 4 個完整頁面，再逐步擴充，而非一次性承諾 36 頁但只交付 4 頁。
